## 一、项目概述

### 1.1 项目背景与意义

随着零售行业的发展，传统的人工记账方式已经无法满足超市日常管理的需求。本项目设计并实现了一个基于 Web 的超市管理系统，旨在帮助中小型超市实现商品管理、销售收银、进货管理、会员管理等业务的信息化和自动化。

### 1.2 项目目标

- 实现商品信息的增删改查管理
- 实现销售收银流程，自动扣减库存
- 实现进货入库流程，自动增加库存
- 实现会员管理与自动积分
- 实现供应商信息管理
- 实现经营数据的可视化展示（仪表盘）
- 实现退换货管理流程
- 实现操作日志记录和审计

### 1.3 系统功能模块

```
超市管理系统
├── 首页仪表盘        — 经营数据总览
├── 商品管理          — 商品信息 CRUD
├── 分类管理          — 商品分类 CRUD
├── 销售收银          — 购物车 + 结算 + 扣库存 + 积分
├── 进货管理          — 进货单 + 入库 + 更新成本
├── 会员管理          — 会员信息 + 积分 + 消费记录 + 等级
├── 供应商管理        — 供应商信息 CRUD
├── 退换货管理        — 退单创建 + 审核 + 库存恢复
├── 库存预警          — 库存不足商品提醒
├── 报表统计          — 销售/库存/利润数据分析
└── 操作日志          — 关键操作记录审计
```

> **CRUD** 是 Create（增）、Read（查）、Update（改）、Delete（删）的缩写，是数据库操作的四种基本操作。

## 二、技术架构详解

### 2.1 整体架构

本系统采用 **前后端分离** 的 B/S（Browser/Server，浏览器/服务器）架构：

```
┌─────────────────────────────────────────────────────┐
│                    用户浏览器                         │
│              （Edge / Chrome 等）                     │
└──────────────────────┬──────────────────────────────┘
                       │ HTTP 请求（端口 5173）
                       ▼
┌─────────────────────────────────────────────────────┐
│              前端（Vite 开发服务器）                   │
│         Vue 3 + Element Plus + Pinia + Vue Router    │
│         负责页面展示、用户交互、发送 API 请求          │
└──────────────────────┬──────────────────────────────┘
                       │ API 请求（/api/*）
                       │ 通过 Vite 代理转发到后端
                       ▼
┌─────────────────────────────────────────────────────┐
│              后端（Express 服务器）                    │
│           Node.js + Express.js                       │
│           负责处理业务逻辑、操作数据库                  │
└──────────────────────┬──────────────────────────────┘
                       │ SQL 查询
                       ▼
┌─────────────────────────────────────────────────────┐
│              数据库（SQLite）                          │
│           supermarket.db 文件                         │
│           存储所有业务数据                              │
└─────────────────────────────────────────────────────┘
```

## 三、数据库设计

### 3.1 数据库选型

本系统使用 **SQLite** 数据库，原因如下：
- **零配置**：无需安装数据库服务，数据存储在一个 `.db` 文件中
- **轻量级**：整个数据库就是磁盘上的一个文件，方便备份和迁移
- **性能够用**：对于中小型超市的数据量（几万条记录），SQLite 完全胜任
- **开发简单**：无需配置连接池、用户名密码等

### 4.3 关键业务逻辑

#### 创建销售单的流程（POST /api/sales）

```
1. 接收请求数据（会员ID、支付方式、商品列表）
2. 计算总金额 = Σ(每个商品的 数量 × 单价)
3. 检查每个商品的库存是否充足
   → 库存不足则返回错误提示
4. 查询会员等级，获取折扣率和积分倍率
5. 应用折扣，计算最终金额
6. 创建销售单记录（写入 sales 表）
7. 遍历商品列表，逐个：
   a. 写入销售明细（写入 sale_items 表）
   b. 扣减商品库存（UPDATE products SET stock = stock - 数量）
8. 如果有会员：
   a. 增加积分（积分 = 最终金额 × 积分倍率）
   b. 增加累计消费金额
   c. 自动升级会员等级
9. 记录操作日志
10. 返回销售单号和总金额
```

> 以上步骤使用 **数据库事务（Transaction）** 保证原子性：要么全部成功，要么全部失败回滚，不会出现扣了库存但没记录销售单的情况。

#### 创建进货单的流程（POST /api/purchases）

```
1. 接收请求数据（供应商ID、商品列表）
2. 计算总金额 = Σ(每个商品的 数量 × 进货单价)
3. 创建进货单记录
4. 遍历商品列表，逐个：
   a. 写入进货明细
   b. 增加商品库存（UPDATE products SET stock = stock + 数量）
   c. 更新商品成本价为最新进货价
5. 记录操作日志
6. 返回进货单号和总金额
```

#### 创建退单的流程（POST /api/returns）

```
1. 接收请求数据（原订单号、退货商品列表、退货原因）
2. 验证原订单存在
3. 验证退货数量不超过原购买数量（减去已退数量）
4. 计算退款金额
5. 创建退单记录（状态为"待审核"）
6. 创建退货明细记录
7. 记录操作日志
```

#### 审核退单的流程（PUT /api/returns/:id/approve）

```
1. 验证退单存在且状态为"待审核"
2. 如果是"拒绝"操作：
   a. 更新退单状态为"已拒绝"
   b. 记录操作日志
3. 如果是"通过"操作：
   a. 更新退单状态为"已完成"
   b. 恢复库存（增加商品库存）
   c. 如果有会员，扣减积分和累计消费
   d. 记录操作日志
```

- 自动注入 token，统一错误处理

#### 组件复用
- CrudTable：通用表格组件，封装搜索、分页、操作
- CrudDialog：通用弹窗组件，封装表单、验证、提交
- 大幅减少重复代码，提高开发效率

### 5.4 前端与后端的交互流程

以"查看商品列表"为例：

```
1. 用户点击左侧菜单"商品管理"
2. Vue Router 匹配路由 /products，加载 Products.vue 组件
3. 组件挂载时（onMounted），调用 productsApi.getProducts()
4. Axios 发送 GET 请求到 /api/products
5. Vite 开发服务器将请求代理转发到 http://localhost:3000/api/products
6. Express 后端接收到请求，查询 SQLite 数据库
7. 数据库返回商品数据
8. Express 将数据以 JSON 格式返回给前端
9. Axios 接收到响应数据
10. Vue 将数据赋值给响应式变量 products
11. 页面上的 el-table 组件自动渲染商品列表
```

## 七、系统安全性设计

### 7.1 用户认证

- 使用 JWT（JSON Web Token）进行用户认证
- 密码使用 bcrypt 算法加密存储
- Token 有效期 24 小时，过期自动跳转登录

- 基于角色的访问控制（RBAC）
- 两种角色：admin（管理员）、cashier（收银员）
- 前端路由守卫 + 后端中间件双重校验
- 管理员可访问所有功能，收银员部分功能受限

### 7.3 数据安全

- 使用参数化查询防止 SQL 注入
- 软删除策略，数据可恢复
- 操作日志记录所有关键操作
- 数据库文件定期备份

## 八、答辩常见问题及参考回答

### Q1: 请介绍一下你的项目

> 这是一个基于 Web 的超市管理系统，采用前后端分离架构。前端使用 Vue 3 配合 Element Plus 组件库和 Pinia 状态管理，后端使用 Node.js 的 Express 框架，数据库使用 SQLite。系统实现了商品管理、分类管理、销售收银、进货管理、会员管理、供应商管理、退换货管理、报表统计、操作日志等功能模块，以及一个数据可视化仪表盘。用户通过浏览器访问系统，可以完成超市日常运营中的各项管理操作。

### Q2: 为什么选择这些技术？

> **前端选择 Vue 3 + Element Plus + Pinia**：Vue 是国内使用最广泛的前端框架，中文文档完善，社区活跃。Element Plus 是专为 Vue 3 设计的 UI 组件库，提供了表格、表单、对话框、菜单等管理系统常用的组件，可以大幅提高开发效率。Pinia 是 Vue 官方推荐的状态管理方案，响应式数据更新，组件自动同步。
>
> **后端选择 Node.js + Express**：Node.js 让 JavaScript 可以同时在前端和后端使用，降低了技术栈的复杂度。Express 是 Node.js 生态中最成熟的 Web 框架，轻量灵活，适合中小型项目。
>
> **数据库选择 SQLite**：对于超市管理这种数据量级（几千到几万条记录），SQLite 完全够用。它的优势是零配置、无需安装数据库服务、数据存储在一个文件中方便备份和迁移。如果未来数据量增大，也可以很方便地迁移到 MySQL。

### Q3: 什么是前后端分离？为什么要这样做？

> 前后端分离是指前端和后端作为两个独立的部分分别开发和部署。前端负责页面展示和用户交互，后端负责业务逻辑和数据处理，两者通过 API（应用程序编程接口）进行数据交换。
>
> 好处：
> 1. **职责清晰**：前端专注 UI，后端专注逻辑，分工明确
> 2. **开发效率高**：前后端可以并行开发，互不阻塞
> 3. **可维护性强**：修改前端不影响后端，修改后端不影响前端
> 4. **可扩展性好**：同一个后端可以同时服务网页、手机 App、小程序等多种客户端

### Q4: 前后端是如何通信的？

> 前端通过 HTTP 请求调用后端提供的 RESTful API 接口。具体流程：
> 1. 前端使用 Axios 库发送 HTTP 请求（如 GET /api/products）
> 2. 在开发环境下，Vite 开发服务器会将 `/api` 开头的请求代理转发到后端的 3000 端口
> 3. 后端 Express 接收请求，执行对应的路由处理函数
> 4. 路由函数操作 SQLite 数据库，查询或修改数据
> 5. 后端将结果以 JSON 格式返回给前端
> 6. 前端接收 JSON 数据，渲染到页面上

### Q5: 什么是 RESTful API？

> RESTful 是一种 API 设计风格，核心思想是用 HTTP 方法表示操作类型：
> - **GET** 表示查询（读取数据）
> - **POST** 表示新增（创建数据）
> - **PUT** 表示修改（更新数据）
> - **DELETE** 表示删除（删除数据）
>
> URL 表示操作的对象，比如 `/api/products` 表示商品资源，`/api/products/1` 表示 ID 为 1 的商品。这样设计的 API 语义清晰、结构统一、易于理解和维护。

### Q6: 销售时如何保证库存数据的准确性？

> 使用了数据库的 **事务（Transaction）** 机制。在创建销售单时，扣减库存、写入销售记录、更新会员积分这些操作被放在一个事务中。事务保证这些操作要么全部成功，要么全部失败回滚。这样就不会出现"库存扣了但销售记录没写入"或"销售记录写了但库存没扣"的数据不一致问题。
>
> 在代码中使用了 `db.transaction()` 包裹所有操作，SQLite 会自动处理事务的提交和回滚。

### Q7: SQLite 和 MySQL 有什么区别？为什么不用 MySQL？

> | 对比项 | SQLite | MySQL |
> |--------|--------|-------|
> | 安装 | 无需安装，一个文件 | 需要安装数据库服务 |
> | 配置 | 零配置 | 需要配置用户、密码、端口 |
> | 性能 | 适合中小数据量 | 适合大数据量、高并发 |
> | 部署 | 复制文件即可 | 需要安装和配置服务 |
> | 适用场景 | 单机应用、小型项目 | 企业级应用、网站 |
>
> 本系统面向中小型超市，数据量在几万条以内，SQLite 完全够用，且部署简单、无需额外安装。如果未来需要扩展，可以很方便地迁移到 MySQL。

### Q8: Vue 的响应式数据是什么意思？

> Vue 的响应式是指：当数据发生变化时，页面会自动更新，无需手动操作 DOM。
>
> 比如在商品管理页面中，`products` 是一个响应式变量（用 `ref` 定义），它绑定到了表格组件上。当后端返回新的商品数据并赋值给 `products` 时，Vue 会自动检测到变化，并重新渲染表格，显示最新的数据。
>
> 这就是 Vue 的核心优势——开发者只需关注数据，不需要手动操作页面元素。

### Q9: Element Plus 在项目中起了什么作用？

> Element Plus 是一个基于 Vue 3 的 UI 组件库，提供了大量预设好的界面组件。在本项目中使用了：
> - **el-table**：表格组件，用于展示商品列表、会员列表等数据
> - **el-form / el-input**：表单组件，用于新增和编辑数据的输入
> - **el-dialog**：对话框组件，用于弹出编辑窗口
> - **el-menu**：菜单组件，用于左侧导航栏
> - **el-card**：卡片组件，用于仪表盘的统计卡片
> - **el-message / el-message-box**：消息提示组件
> - **el-tag**：标签组件，用于显示状态（如库存预警）
>
> 使用组件库的好处是不需要自己写 CSS 样式，开箱即用，界面美观统一。

### Q10: 项目中遇到了什么困难？怎么解决的？

> 可以从以下几个方向回答（选择适合自己的）：
>
> 1. **Vite 与 Node.js 版本兼容问题**：最新版 Vite 7 使用了 rolldown 原生模块，与 Node.js v24 不兼容。通过降级到稳定的 Vite 5 解决。
> 2. **数据库事务处理**：销售时需要同时操作多张表（sales、sale_items、products、members），需要保证数据一致性。通过使用 SQLite 的事务机制解决。
> 3. **前后端跨域问题**：开发时前端在 5173 端口，后端在 3000 端口，浏览器会阻止跨域请求。通过 Vite 的代理配置和后端的 cors 中间件解决。
> 4. **代码复用问题**：多个页面有相似的表格和弹窗逻辑，导致大量重复代码。通过抽象通用组件（CrudTable、CrudDialog）解决。
> 5. **状态管理问题**：用户信息散落在 localStorage，组件间状态不同步。通过引入 Pinia 状态管理解决。

### Q11: 这个系统还有什么可以改进的地方？

> 1. **数据报表导出**：可以增加将销售报表导出为 PDF 的功能
> 2. **条码扫描**：可以对接扫码枪硬件，实现扫码收银（已支持）
> 3. **数据持久化与云部署**：将数据库迁移到 MySQL，部署到云服务器，支持多终端访问
> 4. **打印小票**：对接打印机，销售完成后自动打印小票（已支持）
> 5. **库存盘点**：增加库存盘点功能，支持实际库存与系统库存的对比
> 6. **多门店管理**：支持多个门店的数据隔离和统一管理
> 7. **移动端适配**：开发手机 App 或小程序版本

### Q12: 系统的安全性如何考虑？

> 1. **用户认证**：使用 JWT（JSON Web Token）进行无状态认证
> 2. **密码加密**：使用 bcrypt 算法对密码进行哈希加密存储
> 3. **权限控制**：基于角色的访问控制（RBAC），前后端双重校验
> 4. **SQL 注入防护**：使用参数化查询，防止 SQL 注入攻击
> 5. **操作日志**：记录所有关键操作，便于审计和追踪
> 6. **数据备份**：SQLite 数据库文件可以方便地备份和恢复
> 7. **软删除策略**：数据采用软删除，误删可恢复
